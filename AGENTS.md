## Repository Purpose

`talos-platform-apps` is the **central platform catalog** of the Devoba Talos platform. **Everything that is not substrate** (not in `talos-platform-base`) lives here and is published as signed OCI artifacts. Consumer-cluster repos **draw from the catalog** ‚Äî they reference exactly the components they need by tag. Division of labor: **Base = substrate (Talos + Cilium + ArgoCD + cert-approver), Apps = catalog (everything else), Consumer = composition.**

**Granularity**: the OCI distribution unit is the **component**; the sub-layer is an organizational grouping (directory grouping). See ADR-0009 ¬ß OCI granularity. Sub-layers as a grouping (as of 2026-06-03, taxonomy decision #16): `automation`, `databases`, `lifecycle`, `observability` (formerly `monitoring`), `registry`, `secrets`, `storage-objects` plus the capability-driven `identity`, `network`, `compute`, `storage-block`, `security` ‚Äî within each sub-layer 1-N components live as independently versioned OCI artifacts. `kube-prometheus-stack` is a *stack* (a composition of individual apps), not a component of its own.

**NOT** in this repo:

- Cluster identity (node IPs, hostnames, TLS cert CNs)
- Real secrets (Vault tokens, age keys, passwords)
- Cluster-specific Helm overrides (belong in the consumer-cluster repos)
- Runnable clusters or kubeconfigs

## Build, Test, Development Commands

Prerequisite: Devbox + direnv. After `direnv allow` all tools are on `PATH`.

| Command | Purpose |
|---|---|
| `task lint` | YAML lint over `sub-layers/`, `.github/`, `Taskfile.yml`; Markdown lint over **all tracked `.md`** (`git ls-files '*.md'`, excluding `CHANGELOG.md`) |
| `task lint:rendered` | `kubeconform` over all components' `rendered/` outputs |
| `task render:one -- <sub-layer>/<component>` | `helm template` for one component |
| `task render:sublayer -- <sub-layer>` | renders all components of one sub-layer |
| `task render` | render of all components of all sub-layers (matrix) |
| `task push -- <sub-layer>/<component> <tag>` | `oras push` to `oci://.../<sub-layer>/<component>:<tag>` (native OCI, single-layer tarball = a Kustomize base: `kustomization.yaml` + `manifest.yaml`, so `sourceType=Kustomize` and consumers overlay via `source.kustomize.patches`) |
| `task sign -- <sub-layer>/<component> <tag>` | `cosign sign --yes` (local registries ‚Üí skip) |
| `task attest -- <sub-layer>/<component> <tag>` | SBOM (syft ‚Üí CycloneDX ‚Üí cosign attest) + SLSA provenance ‚Äî phase 2+ |
| `task publish -- <sub-layer>/<component> <tag>` | render ‚Üí package ‚Üí push ‚Üí sign in one go |
| `task publish:sublayer -- <sub-layer> <tag>` | publish of all components of one sub-layer with the same tag |
| `task ci` | **local reproduction of the GHA pipeline** (all components, lint + render + conftest, no push) |
| `task worktree:create -- <sub-layer>/<component>` | create an isolated git worktree (`.claude/worktrees/<slug>`, branch `catalog-build/<slug>`) for parallel independent sessions; cross-session-safe (`mkdir` lock), idempotent, branch = claim; prints the path |
| `task worktree:remove -- <sub-layer>/<component>` | remove the worktree (the branch is kept) |
| `task worktree:list` | list active component worktrees |

**Never use `make`** ‚Äî the convention is go-task.

**Taskfile conventions**:

- `silent: true` globally ‚Äî commands do not echo themselves
- **Logic lives in the Taskfile**, not in external `scripts/` bash files. Even complex bash code is implemented inline in `cmds:` (multi-line `|`). External scripts would undermine the "pipeline = task caller" convention (pipeline ‚Üí task ‚Üí script would be one level too deep).

## Commits & Pull Requests

- **Conventional Commits** with sub-layer or component scope: `feat(lifecycle): ‚Ä¶`, `feat(lifecycle/crossplane): ‚Ä¶`, `fix(secrets): ‚Ä¶`, `chore(automation): ‚Ä¶`, `docs: ‚Ä¶`
- One commit = one logical unit. Never commit render output (`rendered/`).
- **PR-title + single-component gate** (`commit-lint.yml` ‚Üí `task lint:pr`, issue #464): every PR to `main` is checked so the **PR title** is a Conventional Commit **and** the **whole-PR diff touches at most one component**. Under squash-only merges the PR title becomes the squash commit subject that release-please parses, and a multi-component PR collapses to one squash commit that would version-bump every touched component ‚Äî so this is the squash-time enforcement of the per-commit `lint:commit-scope` (which a squash evades). A multi-component change is **one PR per component**. Promoting `commit-lint` to a **required** status check and the squash-only repo settings (`squash_merge_commit_title=PR_TITLE`) are armed by the branch-protection step in #463.
- **Signed commits are mandatory**: `main` enforces branch-protection `required_signatures`. An unsigned commit makes the PR `mergeStateStatus: BLOCKED`; the fix is to sign, never to admin-override the gate. Configure commit signing **once, globally** on your machine (`git config --global gpg.format ssh` / `user.signingkey <your-public-key.pub>` / `commit.gpgsign true`) ‚Äî a per-machine identity setting that covers every repo, clone, and worktree, so there is no per-clone step. See `README.md` ¬ß Commit signing for the GitHub-side steps (register the key as a *Signing* key, verify the committer email). Distinct from cosign OCI artifact signing. If the signing key is passphrase-protected it MUST be loaded into the ssh-agent, otherwise non-interactive / agent-driven commits fail to sign.
- **Breaking-change bumps**: a new major tag (`<sub-layer>/<component>-v2.0.0`) requires a `BREAKING CHANGE:` footer in the commit and an entry in the top-level `CHANGELOG.md` (if present).
- PR body: what changed + why + validation steps (see `.github/PULL_REQUEST_TEMPLATE.md`).
- PRs should go through **subagent reviews** (see Multi-Agent Coordination below). The `require-review.sh` hook is **deliberately inactive** (not bound in `settings.json`) until M2 onboards ‚Äî with a single maintainer, fail-closed would be self-sabotage. The hook scripts stay in the repo and are reactivated once a multi-maintainer workflow becomes real.

## Branch protection & merge gates

`main` is protected; a PR merges only when every gate below is green. **The live GitHub branch-protection config is authoritative** ‚Äî this section describes the configured contract so every contributor and every agent harness knows how a PR becomes mergeable *without querying the API*. If the two ever diverge, the live config wins and this section MUST be corrected. **Never `gh pr merge --admin`**: `enforce_admins` is off, so an admin *can* bypass, but bypassing a gate is a policy violation, not a workflow ‚Äî this is a multi-maintainer repo (see ¬ß Multi-Agent Coordination + `.github/CODEOWNERS`). The fix for a `mergeStateStatus: BLOCKED` PR is always to satisfy the gate.

**Merge method ‚Äî squash-only.** Merge-commit and rebase merges are disabled; `gh pr merge <N> --squash` is the only path (`--merge` / `--rebase` are rejected). The squashed commit takes its **subject from the PR title** (`squash_merge_commit_title=PR_TITLE`) and its **body from the PR description** (`squash_merge_commit_message=PR_BODY`). Consequence: the **PR title MUST be a valid Conventional Commit with a single sub-layer/component scope** ‚Äî that title becomes the commit release-please path-maps to a component (¬ß CI conventions ‚Üí Release automation). `required_linear_history` keeps `main` merge-commit-free, consistent with squash-only.

**Required status checks** ‚Äî all MUST be green, and `strict` is on so the PR branch MUST also be up to date with `main`:

| Check (context name) | Enforces |
|---|---|
| `ci` | `task ci` ‚Äî render + kubeconform + conftest (ADR-0018) + `validate:contract` + `validate:crd-split` (ADR-0028) + `validate:release-config` |
| `validate-contract` | component-contract schema conformance |
| `require-issue-link` | the PR links an issue (`Closes #N`) **or** carries the `no-issue` label (`pr-issue-link.yml`) |
| `gitleaks (secret-scan)` | no secret leaks in the PR's changed commit range (`security-scan.yml`) ‚Äî the context name is the job name `gitleaks (secret-scan)`, **not** bare `gitleaks` |
| `commit-lint` *(pending)* | Conventional PR title + single-component scope (G2, #464/#465); added as a required context once #465 merges |

**Non-status-check merge gates:**

- **Signed commits** (`required_signatures`) ‚Äî an unsigned commit ‚Üí `BLOCKED`; see ¬ß Commits & Pull Requests for the one-time setup.
- **CODEOWNERS review** ‚Äî ‚â•1 approving review from the code owner (`.github/CODEOWNERS`; `require_code_owner_reviews`, `required_approving_review_count: 1`; stale reviews are dismissed on a new push).
- **Conversation resolution** (`required_conversation_resolution`) ‚Äî all PR review threads MUST be resolved.

**Branch integrity.** Force-pushes and branch deletion on `main` are disabled (`allow_force_pushes: false`, `allow_deletions: false`).

**Not a PR merge gate (publish-time, advisory today).** The image-CVE control runs *post-merge* in `oci-publish.yml` (tag-triggered), currently advisory; the PR-level `trivy-cve` scan in `security-scan.yml` is deliberately **not** a required check. See ¬ß ADR-Abdeckung (ADR-0018) ‚Äî it becomes a hard block after the E5 flip.

## CI conventions (binding)

Three rules, project-wide:

1. **Devbox cache active** ‚Äî every GHA job uses `jetify-com/devbox-install-action@v0.x` with `enable-cache: true`. Tool versions come from `devbox.json`/`devbox.lock`. **Never `actions/setup-go`, `actions/setup-helm`, etc.**
2. **Locally reproducible** ‚Äî every task runs on the workstation exactly as in CI. Before `git push`, `task ci` runs locally. No GHA-specific code in tasks (`$GITHUB_ACTIONS` checks or similar are forbidden).
3. **Pipeline = task caller** ‚Äî workflow steps only call `task <name>`. No inline `helm template`/`oras push`/`cosign sign` commands in the YAML. You change behavior in the task, not in the workflow.

**Tooling carve-out (rule 1 only; closed set ‚Äî one entry).** `openknowledge` (the OKF-bundle validator for `knowledge/`, gated via `task okf:validate` in `task ci`) is **not** packaged in Nixpkgs, so it is the single tool NOT sourced from `devbox.json`. `task okf:install` installs it from a **version-pinned GitHub release asset** whose per-platform sha256 is pinned in the Taskfile; it is idempotent, checksum-verifies before extraction (no curl-pipe-to-shell, retry + timeout on the download), and runs identically locally and in the GHA `ci` job (called by `task ci`), so rule 2 (locally reproducible) still holds. The `ci` job caches `.okf/` (keyed on the Taskfile hash) so a normal PR restores the CLI rather than re-downloading. A version bump re-pins every platform checksum in the Taskfile together. This is a closed one-entry list ‚Äî a second non-devbox tool joins it only by amending this note. **Accepted supply-chain residual**: upstream ships only `checksums.txt` ‚Äî no cosign signature, no SLSA provenance ‚Äî so the sha256 pin detects post-adoption asset tampering but not a pin-time upstream compromise; this is asymmetric with the repo's own cosign-keyless OCI signing and accepted for a build-time doc validator.

**API-orchestration carve-out (rule 3 only; closed set).** Two workflow classes are GitHub-API orchestration ‚Äî *not* build-pipeline steps ‚Äî and are therefore exempt from rule 3 (they cannot be locally reproduced):

- **Label / board automation** (inline `actions/github-script`, no devbox/`task`): `issue-triage.yml`, `project-sync.yml`, `pr-needs-review.yml`, `status-strip.yml`, and `pr-issue-link.yml` (a PR‚Üíissue-link gate keeping the Projects v2 board self-maintaining ‚Äî a linked PR auto-closes its issue on merge, so Status=Done needs no manual upkeep; a PR closing no issue must carry the `no-issue` label).
- **Release management**: `release-please.yml` ‚Äî two `uses:` actions only (`actions/create-github-app-token` + `googleapis/release-please-action`), no build steps. It cuts the release tag; the build/publish runs through `oci-publish.yml` (the `task` caller), triggered by that tag.
- **Security-scan escalation** (step-level): `security-scan.yml`'s `trivy-cve-all` weekly job opens/updates a `risk: high` issue on findings via inline `actions/github-script` (AC7). Only that escalation **step** is the carve-out ‚Äî the scan itself is `task scan:trivy-cve-all` (rule-3 compliant), and the advisory `trivy-cve` PR job is a pure `task` caller.

This is a closed enumeration, not a self-applied "it's automation" escape hatch ‚Äî a new workflow joins it only by amending this list. All listed workflows are still linted by rule 1/2's `yamllint` (`task lint` lints `.github/`).

**Release automation (binding).** Component releases are managed by **release-please** (`release-please.yml`), not hand-cut tags. A `feat`/`fix` commit under `sub-layers/<sl>/components/<c>/` (one component per commit ‚Äî enforced by the lefthook `lint:commit-scope` gate, because release-please maps commits to components path-based) makes release-please open a per-component release PR (`separate-pull-requests`); merging it cuts the tag `<sub-layer>/<component>-vX.Y.Z`. **Mass-release deadlock prevention (binding):** every per-component release PR edits the *same* `.release-please-manifest.json`, so a bulk release (e.g. a migration flood opening dozens of release PRs at once) makes the adjacent JSON lines conflict ‚Äî once some siblings merge, the rest go `DIRTY` and, at default `always-update: false`, release-please never rebases a PR whose own release notes did not change, so nothing merges ‚Üí no `push:[main]` ‚Üí release-please never re-runs ‚Üí self-reinforcing stall. Mitigated (not root-cause-eliminated ‚Äî the single shared manifest is inherent to release-please manifest mode): `always-update: true` + `sequential-calls: true` (config) make each run recreate **every** open release PR on current main HEAD (sequentially, to dodge GitHub secondary-rate-limit throttling under a mass rebase), clearing the textual conflict ‚Äî so an ordinary merge rebases the stuck PRs via a serial cascade instead of leaving them `DIRTY`; `workflow_dispatch` on `release-please.yml` is the escape hatch for the one state a merge cannot reach (all open release PRs `DIRTY`, no mergeable PR left to push main). **This is not fully autonomous under a mass wave.** Because branch protection sets `dismiss_stale_reviews: true`, each rebase-push dismisses that PR's approval, so a flood drains as **O(n) serial re-approvals** ‚Äî release-please does the rebase, the human only re-approves (no git rebuild, unlike the pre-fix manual break); approve one at a time as each merges (approve-all-upfront wastes approvals to the next dismissal). Steady-state cost is non-zero but mild (any unrelated merge to main dismisses open release-PR approvals, but there is rarely >1 open). A **full-catalog wave** (all ~60 components at once) can also exceed the workflow timeout mid-sweep; it is a known manual-intervention event ‚Äî re-run via `workflow_dispatch`, and successive sweeps finish faster as the queue drains. The tag is created with a **GitHub App installation token** (not the built-in `GITHUB_TOKEN`, whose events do not cascade to other workflows), so the tag-push triggers `oci-publish.yml`, which renders + pushes + signs + verifies. The cosign signing identity therefore stays `oci-publish.yml@refs/tags/...` ‚Äî unchanged for `task verify` and for the consumer-side Kyverno policy. The per-component version SoT is `.release-please-manifest.json`; `release-please-config.json` lists every component (stubs carry `initial-version: 0.1.0`); every `components/` directory is a release package, with no exclusions ‚Äî a *stack* like `kube-prometheus-stack` is a composition documented in its sub-layer README, never a `components/` directory, so there is nothing to exclude. A manual tag push still triggers `oci-publish.yml` for backfill/hotfix ‚Äî push **‚â§ 3 tags per `git push`** (tags beyond the third in a single push raise no workflow run at all ‚Äî silently ‚Äî so batch larger backfills). One-time setup: a GitHub App (repo permissions Contents + Pull requests + Issues: write) installed on the repo, with `RELEASE_PLEASE_APP_ID` (variable) + `RELEASE_PLEASE_APP_PRIVATE_KEY` (secret) ‚Äî see `README.md ¬ß Release automation`. **Adoption cutover:** `bootstrap-sha` marks where release-please begins managing this repo; component changes merged *before* it are **not** auto-released by the first run (release-please anchors on its own last release PR or `bootstrap-sha`, never on the manually-cut tags ‚Äî and the manifest already records each component's last published version), so pre-cutover unreleased changes are published via the manual-backfill path when a component next warrants a version (`bootstrap-sha`/`last-release-sha` are top-level-only, so there is no per-package config alternative). Pre-cutover backfill tracker: see issue #259. **`release-please-config.json` ‚Üî component-directory parity is gated by `task validate:release-config` (in `task ci`)** ‚Äî a new component absent from the config (or a stale config entry) fails the pipeline, so a component cannot be silently left out of release management.

## Hard Constraints (universal cluster invariants)

These **MUST NOT** be relaxed without explicit maintainer approval.

- **No direct `kubectl apply` against clusters** from this repo. This repo publishes OCI artifacts; cluster apply runs via Argo in the consumer repos.
- **No real secrets in the repo** ‚Äî not even in tests. `.sops.yaml.tmpl` stays a template until issue #3 delivers the four age recipients.
- **No `make` target** ‚Äî the convention is go-task. Anyone who submits a `Makefile`: review will reject it.
- **Do not commit render output** ‚Äî `rendered/` is gitignored. Checked via the lefthook `no-rendered` pre-commit job.
- **No `.envrc.local`** in the repo ‚Äî direnv local overrides stay local.
- **Do not commit `.claude/` with personalized settings** ‚Äî `settings.local.json` is gitignored.
- **OCI paths hardcoded**: `ghcr.io/devobagmbh/talos-platform-apps/<sub-layer>/<component>:<tag>` ‚Äî renaming the org path requires coordination with all consumers.
- **cosign keyless with GHA OIDC**: the signing identity is the workflow identity. Never commit long-lived keys.

## Sub-layer and component conventions

- **One directory per sub-layer** under `sub-layers/<name>/`, with `components/<component>/` beneath it per OCI artifact. Directory name == OCI-path component.
- **Per sub-layer**: `README.md` (required, lists components + sync-wave order), `compatibility.yaml` (aggregate).
- **Per component**: `README.md` (required: content + OCI path + sync-wave + ADR references), `compatibility.yaml` (required: requires/provides), `customization.yaml` (required: ADR-0024 v2 freeze-line contract, validated against `schemas/customization.schema.json`), `helm/` or `manifests/` (content).
- **Consumer separation**: this repo holds defaults and shared values. Cluster-specific things (replica counts, VIPs, OIDC issuer URLs) belong in the consumer repos.
  - **How a consumer overlays (calibrated-friction, ADR-0024).** Each component is published as a **Kustomize base** (`task package` puts a trivial `kustomization.yaml` referencing `manifest.yaml` into the OCI tarball ‚Üí `sourceType=Kustomize`). The consumer's Argo `Application` overlays per-cluster fields **without a catalog PR** via `source.kustomize.patches`. The base itself restricts nothing ‚Äî a patch can technically touch any field, including the image; the boundaries are enforced **downstream, not by the artifact format**: the image digest is pinned at consumer admission (cosign `verifyImages`), `sync_wave` and other platform-set fields stay catalog-owned, and dangerous classes (hostPath, cluster-admin bindings) are fenced by the consumer-side ADR-0018 Kyverno safe-defaults ‚Äî never a catalog allowlist. Example: raise a workload's memory limit per-cluster.

    ```yaml
    # consumer Application source (consumer repo)
    source:
      repoURL: oci://<registry>/<sub-layer>/<component>
      targetRevision: <tag>
      path: "."
      kustomize:
        patches:
          - target: { kind: Deployment, name: <name> }
            patch: |
              - op: replace
                path: /spec/template/spec/containers/0/resources/limits/memory
                value: 512Mi
    ```

    Migration note: publishing a component in this form flips its ArgoCD auto-detection from `sourceType=Directory` to `Kustomize`. A consumer `Application` relying on `spec.source.directory.{include,exclude,recurse}` on the OCI source must move that logic into the Kustomize layer ‚Äî those options become ineffective once `kustomization.yaml` is present (Argo prefers Kustomize detection). Consumer-side wiring for a **private** OCI registry (the `type: oci` repository Secret, registry CA trust) is a consumer-repo concern, documented there ‚Äî not here.
- **Argo Application definitions live in the consumer-cluster repo**, not here. One `Application` CR per component with an `argocd.argoproj.io/sync-wave` annotation. For local end-to-end tests there are `local/argo-apps/<sub-layer>/<component>.yaml` templates in the apps repo.
- **CRD management ‚Äî strict B** (talos-platform-docs ADR-0028). Every component that ships chart-provided CustomResourceDefinitions publishes a **separate** `<sub-layer>/<component>-crds` OCI artifact next to its workload artifact; the consumer wires **two** Argo `Application`s ‚Äî the `-crds` app at `sync-wave -1` with `Prune=false`, then the workload. No inline CRDs in the workload artifact, no per-component inline-vs-separate choice.
  - A `crd-bearing: true` marker (in `compatibility.yaml` ‚Äî `customization.yaml` is schema-locked `additionalProperties: false` and cannot carry it) drives the split and is the build gate's oracle: `^kind: CustomResourceDefinition` count **> 0** in the `-crds` artifact **and** **== 0** in the workload artifact. Enforced by **`task validate:crd-split`** (part of `task ci`), keyed structurally on the `-crds` suffix + the workload sibling: a `-crds` component must carry the line-anchored marker and render >0 CRDs; its workload half must render 0 (and a non-`-crds` component carrying the marker is flagged as misplaced). The gate validates split **correctness** (a split pair is internally consistent), not split **completeness**: a single-name component that ships chart CRDs inline instead of splitting is not flagged ‚Äî reviewer judgment still gates whether a new CRD-shipping component must split (the out-of-scope operator-installed-CRD case below is one such un-gated class).
  - The build splits deterministically: `yq 'select(.kind == "CustomResourceDefinition")'` over the combined render output (covers both helm- and raw-manifest-sourced CRDs).
  - CR-cascade protection is the **Argo layer** ‚Äî `argocd.argoproj.io/sync-options: Prune=false` on the `-crds` app (authoritative). `helm.sh/resource-policy: keep` is the Helm layer and is **not** honored by Argo for its own prune decisions. `ServerSideApply=true` clears the 262 KB annotation limit on large CRDs.
  - **Out of scope:** operator-installed CRDs (Crossplane providers / XRDs) ‚Äî handled by the existing sync-wave readiness model, not by this convention.
- **`compatibility.yaml` per component** declares the component dependencies:

  ```yaml
  requires:
    # NO talos-platform-base line. apps does NOT depend on the substrate
    # (ADR-0009, #71): base & apps are co-equal inputs the consumer integrates.
    # Declare ONLY catalog-internal component deps + capability requirements;
    # the consumer maps capabilities to concrete base/apps versions.
    lifecycle/crossplane: ">=v0.1.0"   # another component in THIS repo
    cnpg-postgres: "*"                 # a capability (id from catalog/capability-index.yaml)
  provides:
    - name: <component>
      version:
        sot: app                       # provenance axis: app | chart | crd-schema | none
        app: vX.Y.Z                    # version of the running workload
      api_surface: []                  # CRD/API groups exposed (empty for most components)
      capabilities:
        - id: <capability-id>          # if this component implements a capability
  ```

- **The sub-layer `compatibility.yaml`** is an aggregate that lists the components:

  ```yaml
  components:
    - crossplane    # sync-wave 0
    - ipxe          # sync-wave 0
    - providers     # sync-wave 10
    - compositions  # sync-wave 20
  ```

## Multi-Agent Coordination

`.claude/agents/` lists specialized subagents. **5 impl/review roles** (2026-05-26, Bobby's bus-factor critique) ‚Äî with a single maintainer, a differentiated 9-role *review* hierarchy is self-review theater. On top comes **`catalog-evaluator`** as a separate build-time acceptance verifier: this is orthogonal to the review-role count ‚Äî an agent that builds and verifies its own work is the documented self-verification failure (MAST FC3; arXiv:2410.21819 + 2402.08115), hence judge-builder separation instead of convenience. At M2 onboarding, `senior-plan-reviewer`, `principal-architect-reviewer`, `provenance-reviewer`, and `compatibility-reviewer` come back from the git history. The "5" count concerns only the **impl/review** axis; the **plan phase** is a separate axis and adds the pair `catalog-planner` (writes the app plan, no verdict) + `plan-reviewer` (read-only, canonical verdict enum, conformance/adversarial persona per brief) ‚Äî judge-builder separation as with implementation/verify.

| Phase | Agent | Output |
|---|---|---|
| Research (optional) | `researcher` | Findings + sources |
| Implementation | `senior-implementer` | Code diff |
| **Verify (build-time)** (deterministic gate + semantic ACs, separate context) | `catalog-evaluator` | Pass/Fail + findings |
| Security review (Vault/SOPS/cosign/RBAC/policies) | `security-reviewer` | Findings |
| Operational safety (bootstrap/DR/backup) | `operational-safety-reviewer` | Findings |
| **Gate** (triage + approve or block) | `staff-reviewer` | Approve or block |

**Self-review is undesirable even with a single maintainer, but not hard-blocked** ‚Äî switching the agent hat is an anti-drift mechanism, not a substitute for the four-eyes principle. Once M2 is here, the `require-review.sh` hook is reactivated and the full 9-agent model is brought back.

**Catalog ship orchestrator**: the `ship-catalog-app` skill orchestrates the full plan‚Üíapprove‚Üíbuild arc of ONE app in a single session as a thin layer over the two skills below (it does not duplicate their logic). Three invariants: a **mandatory human plan-approval gate** between plan and build (headless ‚Üí stop-after-plan); a **mechanically pre-classified merge gate with the build skill as backstop** ‚Äî dependent components need their dependency merged before the fresh worktree off `origin/main` sees it; ship's conservative pre-check is only a dispatch-saving optimization ahead of this backstop ‚Äî the sole guarantee is **no wrong build** (the authoritative build check stops every inadmissible component); both misclassifications are harmless (over-admission ‚Üí wasted dispatch; false-stall on a dependency merged externally after the fetch ‚Üí transient, resolved by a re-run); and **re-run resumability** from git status, no ship-state file. Spec/DRY source: the SKILL.md (no separate CONVENTIONS.md). For single components without a plan, `build-catalog-component` directly suffices.

**Catalog plan primitive**: upstream, the `plan-catalog-app` skill (with `catalog-planner` + `plan-reviewer`) plans a catalog app through a converging plan‚Üíreview‚Üírevise loop (parallel cross-model personas, finding ledger, hard round cap, explicit termination) into a finding-free plan under `.work/plan/<app>/`. `/build-catalog-component` consumes it per component in `build_order`. Spec: `.claude/skills/plan-catalog-app/CONVENTIONS.md`.

**Catalog build primitives**: for the component issues (#17‚Äì#61), the `build-catalog-component` skill (one component, **per-session unit**) and the `catalog-fleet` workflow (**optional single-operator fan-out**) orchestrate the above chain as builder‚Üíverifier‚Üíreviewer with builder ‚âÝ verifier in separate contexts. **Parallelism runs through independent sessions**: each session builds ONE component in its own git worktree (`task worktree:create` ‚Äî cross-session-safe `mkdir` lock, branch name = claim), so multiple sessions work in parallel on ONE clone; `catalog-fleet` is only for single-operator mass fan-out. Deterministic gate (`task ci` + `task validate:contract` + chart-ref / tamper check) first, LLM judge only for the semantics. Output is branches + report ‚Äî never auto-merge (CODEOWNERS + branch protection). Spec: `.claude/skills/build-catalog-component/CONVENTIONS.md`.

**PR auto-review (`babysit-prs`)**: the `babysit-prs` skill runs ONE hardened auto-review pass over the open PRs and dispatches `pr-gate` per eligible PR, delegating the security-critical withhold decision to the tested `task pr:triage` classifier (fork / self / dirty / `release-please` / governance / `oversized` PRs are withheld to a human). `pr-gate` casts no APPROVE it has not empirically reproduced against the PR head (Phase-4 pre-approval gate). Its default mode posts nothing; autonomous auto-approve is **armed** only by the committed whole-line governance marker below (a plain whole-line fixed-string match against `origin/main`), and even armed each pass needs interactive per-pass consent (headless ‚Üí report). Arming is a deliberate CODEOWNERS-reviewed act ‚Äî because any `AGENTS.md` PR is `governance`-withheld, the loop can never self-arm; with two code owners and `required_approving_review_count: 1`, the arming PR's author (one owner) cannot self-approve, so the other owner's required approval is what makes "accepted by both code owners" true. Spec: `.claude/skills/babysit-prs/SKILL.md`.

## ADR-Abdeckung

Operative obligations on this repo that the sections above do not yet spell out (ADR-0009/0024/0028 are already covered above):

- **`external-secrets` must publish the `GithubAccessToken` generator (CRD) enabled** ([talos-platform-docs ADR-0025](https://github.com/devobagmbh/talos-platform-docs/blob/main/adr/0025-argocd-credentials-no-pat.md)) ‚Äî the consumer-side GHCR-token refresh path depends on it. ESO stays in the apps catalog (`secrets/external-secrets`), never in base.
- **Conftest in apps-CI is *the* pre-OCI-push policy gate** ([talos-platform-docs ADR-0018](https://github.com/devobagmbh/talos-platform-docs/blob/main/adr/0018-policy-stack.md)) ‚Äî PNI dissolves into Conftest here (apps) plus Kyverno on the consumer cluster. This repo owns the Conftest half (`policies/`), enforcing the sub-layer / chart-source / no-inline-secret / capability-selector / gateway-API-only / no-latest-tag / reserved-label / resource-limit / no-privileged policy set before any OCI push. **Two PSA gates are live today**: `base.pod_security_standards` (every declared Namespace carries a valid `enforce` level) and `conformance.pod_security` via `task scan:psa-conformance` (the shipped workloads actually CONFORM to that declared level ‚Äî catching a too-strict label, e.g. `enforce: baseline` on a hostPath-mounting workload, *before* live admission rather than after; apps PR #328). The declared enforce level MUST be the strictest the workload satisfies; a hostPath / host-namespace / privileged / host-port workload forces `privileged` (baseline AND restricted forbid those).
- **Image-CVE gate is the pre-OCI-push vulnerability control** ([talos-platform-docs ADR-0018](https://github.com/devobagmbh/talos-platform-docs/blob/main/adr/0018-policy-stack.md)) ‚Äî the vulnerability half of the policy stack, complementing the Conftest misconfig half. Three points: an **advisory diff-scoped PR-check** (`task scan:trivy-cve` in `security-scan.yml`, **not** a required check ‚Äî early warning, its diff-scope edge cases never block a merge), a **publish-time scan** (`task scan:trivy-images-of` in `oci-publish.yml`, E5) ‚Äî **currently advisory (`continue-on-error`)** while the catalog is remediated component-by-component (baseline: ~90% of images carry FIXABLE HIGH/CRITICAL, so a hard block would strand nearly every publish); the scan fails-closed on findings, so deleting `continue-on-error` flips it to a hard block once the catalog is green. As a workflow **step** (not a required status check) the hard gate avoids the `gh pr merge --admin` bypass a required-check skip allows ‚Äî a direct GHCR push remains an out-of-band path. Third, a **weekly full-catalog sweep** (`task scan:trivy-cve-all`) that renders + scans as SEPARATE steps and opens a `risk: high` issue **only on a real CVE finding** (scanner exit 1, never a render/scan-error exit 2). Extraction is key-based (`task scan:extract-images`, the single regex source): it keeps every `[Ii]mage:`-keyed ref carrying a **tag or `@sha256:` digest** (matching the `base.no_latest_image_tag` predicate `_has_explicit_tag_or_digest`, so a workload image lacking both is already conftest-blocked before publish), and drops untagged non-pullable operator-config bare names, CRD-description prose, and malformed leading-`-` refs. Bound red-green by `task test:trivy-extract` (in `task ci` **and** `task ci:artifact`, so a direct/backfill tag push validates it too). Severity policy is **E1** (block ALL HIGH/CRITICAL, incl. unfixed ‚Äî no `--ignore-unfixed`); the E5 step pins `TRIVY_IGNORE_UNFIXED` empty, so relaxing to block-fixable-only is a CODEOWNERS-reviewed edit, never a silent GHA variable. Accepted risks go in `.trivyignore.yaml` (native `expired_at:` + `statement:`, CODEOWNERS MUST inspect deltas). **Baseline pre-flight (before the hard flip)**: run `task scan:trivy-cve-all` over `main`, remediate/`.trivyignore.yaml`-accept every HIGH/CRITICAL, AND wire a read-only `DOCKERHUB_TOKEN` (else a Docker Hub anonymous-pull rate-limit strands the first hard publish; a stranded transient scan re-runs via `gh run rerun`). **Scope (not a completeness claim)**: the gate scans image refs the apps artifact directly declares under an `[Ii]mage:` key. It does NOT scan operator operands (`RELATED_IMAGE_*: value:`), Crossplane `spec.package` xpkg, nested-map `image:{repository,tag}` splits, non-`[Ii]mage` CR keys, or kubevirt runtime `virt-*` images ‚Äî those are the operator-runtime / consumer-Kyverno layer's job (M2: live-cluster `trivy k8s`). Residuals: `.trivyignore.yaml` needs an entry-shape + `expired_at ‚â§ N days` + scope guard built EARLY in remediation, before the file is populated (else far-future / global-by-CVE-ID suppressions accrue unchecked); Docker Hub 200/6 h anonymous pull under load (`DOCKERHUB_TOKEN`); the weekly issue-dedup marker is spoofable by an insider with issues:write (low); an **untagged** image under a CR `[Ii]mage:` key is dropped by the tag-filter AND not conftest-blocked (CR kinds are outside `base.no_latest_image_tag`'s workload set) ‚Äî consumer-Kyverno covers it (M2: extend `no_latest_image_tag` to CR image-keys); a folded/block-scalar image value or a trailing-punctuation ref is a rare `helm template` extraction edge (silent drop, or a fail-closed false-positive).
- **Capability-first component contracts** ([talos-platform-docs ADR-0021](https://github.com/devobagmbh/talos-platform-docs/blob/main/adr/0021-capability-layer-model.md)) ‚Äî in `compatibility.yaml`, each `provides[]` entry names what it ships (`name:`, mandatory ‚Äî usually the tool/chart, sometimes a provider/API bundle) and *additively* lists the capabilities it implements under `provides[].capabilities[]` (each `{id, swap_class}`; every `id` must exist in `catalog/capability-index.yaml`). The capability id is the stable interface a consumer composes against; `swap_class` quantifies swap cost (5 classes, `catalog/README.md`): `drop-in` and `label-move` (a producer-pod label move) are consumer-invisible, `data-migration` / `rewrite-required` / `consumer-change` are not. A component with no matching capability carries `capabilities: []` ‚Äî permanent for the `-crds` strict-B halves; a *deferred* capability carries `capabilities: []` with a `# TODO` to the defining issue ‚Äî or omits the key entirely (some api-surface-only entries, e.g. `lifecycle/providers`).
