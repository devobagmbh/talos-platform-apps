# Component `storage-block/snapshot-controller-crds`

The **strict-B CRDs artifact** (talos-platform-docs ADR-0028) for the
[snapshot-controller](https://github.com/kubernetes-csi/external-snapshotter). It
ships **only** the six external-snapshotter CustomResourceDefinitions — the
controller workload is a **separate** component,
[`storage-block/snapshot-controller`](../snapshot-controller/README.md). The two
together form the strict-B pair: CRDs first (this artifact, sync-wave -1), workload
after (sync-wave 0).

The CRDs are rendered from the piraeus.io
[`snapshot-controller`](https://piraeus.io/helm-charts/) Helm chart at version
**5.0.4** (appVersion **v8.5.0**, kubernetes-csi/external-snapshotter). The chart
renders its CRDs as regular Helm-managed resources gated by `installCRDs`, so this
artifact sets `installCRDs: true` and disables every other chart component
(`controller.enabled: false`, `webhook.enabled: false`,
`webhook.tls.autogenerate: false`) to produce a deterministic CRDs-only render.

## What ships

Exactly six `CustomResourceDefinition` resources — three in the
`snapshot.storage.k8s.io` group (served `v1`, storage `v1`) and three in the
`groupsnapshot.storage.k8s.io` group (served `v1beta1` + `v1beta2`, storage
`v1beta2`):

- `volumesnapshotclasses.snapshot.storage.k8s.io` (kind `VolumeSnapshotClass`)
- `volumesnapshots.snapshot.storage.k8s.io` (kind `VolumeSnapshot`)
- `volumesnapshotcontents.snapshot.storage.k8s.io` (kind `VolumeSnapshotContent`)
- `volumegroupsnapshotclasses.groupsnapshot.storage.k8s.io` (kind `VolumeGroupSnapshotClass`)
- `volumegroupsnapshots.groupsnapshot.storage.k8s.io` (kind `VolumeGroupSnapshot`)
- `volumegroupsnapshotcontents.groupsnapshot.storage.k8s.io` (kind `VolumeGroupSnapshotContent`)

No Deployment, no Service, no RBAC, no Namespace, no webhook, no Secret — the
artifact is purely the six CRDs. The snapshot-controller Deployment and its RBAC
ship in the workload artifact `storage-block/snapshot-controller`, not here. The
`webhook.tls.autogenerate: false` pin is REQUIRED in addition to
`webhook.enabled: false`: without it the chart still renders a non-deterministic
autogenerated TLS Secret (`<release>-conversion-webhook`) that would leak into this
artifact.

The `groupsnapshot.storage.k8s.io` CRDs are inert until a consumer authors
group-snapshot CRs; they ship together with the snapshot CRDs because the chart
provides no per-group CRD toggle.

Consumers author `VolumeSnapshotClass` / `VolumeSnapshot` (and optionally the
group-snapshot) CRs against these schemas. Those CRs are **consumer-owned** (they
live in the consumer-cluster repo overlay, not in this catalog component); this
artifact only establishes the CRD schemas so those CRs have a registered type.

## Strict-B consumer wiring (ADR-0028)

The consumer cluster repo wires **two** Argo `Application`s — this `-crds` app
**before** the workload:

1. **`storage-block/snapshot-controller-crds`** Application at
   `argocd.argoproj.io/sync-wave: "-1"` with:

   ```yaml
   argocd.argoproj.io/sync-options: Prune=false,ServerSideApply=true
   ```

   - `Prune=false` is the authoritative CR-cascade protection: it stops Argo from
     deleting the CRDs (and cascading any live `VolumeSnapshot` /
     `VolumeSnapshotContent` CRs) when the source removes them. The Helm-layer
     `helm.sh/resource-policy: keep` is **not** honored by Argo for its own prune
     decisions, so `Prune=false` carries it.
   - `ServerSideApply=true` avoids the 262 KB client-side last-applied annotation
     limit (the snapshot CRD schemas are large) and is the convention for the
     strict-B `-crds` apps.

2. The workload Application **`storage-block/snapshot-controller`** at sync-wave 0,
   which then comes up against CRDs that already exist.

## crd-bearing pairing

This artifact carries `crd-bearing: true` in `compatibility.yaml` — the strict-B
marker and the build gate's oracle (`kind: CustomResourceDefinition` count **> 0**
here, **== 0** in the workload artifact). Its workload counterpart is
`storage-block/snapshot-controller`.

## Upgrading the CRD schema

When this artifact is bumped to a newer external-snapshotter release whose CRD
schemas changed, the consumer's Argo sync applies the new schema in-place
(ServerSideApply). Because the consumer app runs `Prune=false`, fields the upstream
removes are **not** auto-pruned from the cluster; removal needs manual intervention.
A version bump is a separate reviewed change. **Never** toggle `Prune=true` on the
`-crds` Application while live snapshot CRs exist: Argo would cascade-delete those
CRs.

## Capability

api-surface-only, **no capability** — `capabilities: []`. The six snapshot CRDs are
the API surface (schema), not a swappable operational capability. The
snapshot-controller is the sole SIG-Storage reference implementation of the
`snapshot.storage.k8s.io` API — there is no swappable alternative, so no capability
id is defined (precedent: `storage-block/piraeus-operator-crds`,
`observability/prometheus-operator-crds`, and `storage-objects/garage-crds`,
likewise api-surface-only). This is a permanent design state, not a deferral.

## Sync-wave

`-1` — the CRDs land before the snapshot-controller workload at wave 0.
