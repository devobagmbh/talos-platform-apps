# Component `storage-block/snapshot-controller`

The **workload half** (talos-platform-docs ADR-0028, strict-B) of the
[snapshot-controller](https://github.com/kubernetes-csi/external-snapshotter)
pair. It ships the cluster-singleton **snapshot-controller Deployment** â€” the
SIG-Storage controller that watches `VolumeSnapshot` / `VolumeSnapshotContent`
(and the group-snapshot) objects and drives the per-driver CSI snapshotter sidecar
to create and delete volume snapshots. The six CRDs that define the
`snapshot.storage.k8s.io` + `groupsnapshot.storage.k8s.io` API surface are a
**separate** component,
[`storage-block/snapshot-controller-crds`](../snapshot-controller-crds/README.md)
(sync-wave -1); this workload carries **0 CRDs**.

The workload is rendered from the piraeus.io
[`snapshot-controller`](https://piraeus.io/helm-charts/) Helm chart at version
**5.0.4** (appVersion **v8.5.0**, kubernetes-csi/external-snapshotter) with
`installCRDs: false`, `controller.enabled: true`, and the conversion webhook
disabled.

## What ships

A single render of the chart with the values above plus a dedicated Namespace:

- **`snapshot-controller-snapshot-controller` Deployment** â€” the cluster-singleton
  controller (the release name equals the component name `snapshot-controller`, so
  the chart-suffixed Deployment name is `snapshot-controller-snapshot-controller`).
- **ClusterRole + ClusterRoleBinding** â€” the controller is cluster-scoped (it
  watches snapshot objects in every namespace); this cluster-wide RBAC is intrinsic
  to a cluster-singleton snapshot controller and matches upstream.
- **Role + RoleBinding** â€” namespaced leader-election lease RBAC.
- **ServiceAccount** â€” the controller's identity.
- **`snapshot-controller` Namespace** (`manifests/00-namespace.yaml`) â€” PSA
  `enforce: restricted`.

No CRD, no Secret, no conversion-webhook Deployment.

**Cluster-singleton via leader-election.** The controller runs with
`--leader-election=true` (chart default, pinned explicitly in
`helm/snapshot-controller.yaml`). The singleton guarantee is leader-election, NOT
a pinned `replicaCount` â€” `replicaCount` is a consumer-owned cluster property
(ADR-0024) and is deliberately not set here.

**Conversion webhook disabled.** `webhook.enabled: false` together with
`webhook.tls.autogenerate: false`. Both pins are required: without the second one
the chart still renders a non-deterministic autogenerated TLS Secret
(`<release>-conversion-webhook`) that would leak into the workload artifact and
break the freeze-line determinism. A consumer needing multi-version API conversion
can enable the webhook with a cert-manager-issued TLS overlay in their consumer
repo.

## Pod Security Admission

The shipped `snapshot-controller` Namespace carries
`pod-security.kubernetes.io/enforce: restricted` (+ `audit` / `warn`). The
Deployment provably satisfies `restricted`:

- pod-level: `seccompProfile.type: RuntimeDefault` (pinned) + `runAsNonRoot` (chart
  default),
- container-level: `allowPrivilegeEscalation: false` (pinned),
  `capabilities.drop: [ALL]`, `readOnlyRootFilesystem: true`, `runAsUser: 1000`
  (chart defaults),
- no hostPath volume, no host namespace, no host port, no privileged container.

`restricted` is the strictest level the workload satisfies, per the ADR-0018 PSA
conformance rule.

## Strict-B dependency (ADR-0028)

This workload **requires** its CRD sibling
[`storage-block/snapshot-controller-crds`](../snapshot-controller-crds/README.md).
The consumer cluster repo wires **two** Argo `Application`s, the `-crds` app first:

1. **`storage-block/snapshot-controller-crds`** at
   `argocd.argoproj.io/sync-wave: "-1"` with
   `argocd.argoproj.io/sync-options: Prune=false,ServerSideApply=true` â€” so the
   CRDs exist (and are protected from cascade-prune) before any controller pod
   starts.
2. **`storage-block/snapshot-controller`** (this component) at sync-wave `0`, which
   then comes up against CRDs that already exist.

`compatibility.yaml` declares `requires: storage-block/snapshot-controller-crds:
">=v0.1.0"` and `customization.yaml` lists it under `external_dependencies`.

## Consumer enablement path

The snapshot-controller is the cluster machinery; the snapshot *policy* (which
driver, which volumes) is consumer-owned. End to end:

1. **Enable both components.** The consumer adds the
   `storage-block/snapshot-controller-crds` Argo Application (wave -1) and the
   `storage-block/snapshot-controller` Argo Application (wave 0) to their cluster
   composition.
2. **Author a `VolumeSnapshotClass`.** In the consumer-cluster repo overlay, create
   a `VolumeSnapshotClass` naming the CSI driver, e.g.:

   ```yaml
   apiVersion: snapshot.storage.k8s.io/v1
   kind: VolumeSnapshotClass
   metadata:
     name: linstor-snapshots
   driver: linstor.csi.linbit.com
   deletionPolicy: Retain
   ```

3. **Create a `VolumeSnapshot`** of an existing PVC (e.g. a `fast-rwo-r2` PVC):

   ```yaml
   apiVersion: snapshot.storage.k8s.io/v1
   kind: VolumeSnapshot
   metadata:
     name: my-data-snap
   spec:
     volumeSnapshotClassName: linstor-snapshots
     source:
       persistentVolumeClaimName: my-data
   ```

   The snapshot-controller reconciles it into a `VolumeSnapshotContent`.
4. **Restore via a PVC `dataSource`.** Create a new PVC referencing the snapshot:

   ```yaml
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata:
     name: my-data-restored
   spec:
     storageClassName: fast-rwo-r2
     dataSource:
       name: my-data-snap
       kind: VolumeSnapshot
       apiGroup: snapshot.storage.k8s.io
     accessModes: [ReadWriteOnce]
     resources:
       requests:
         storage: 10Gi
   ```

The `VolumeSnapshotClass`, `VolumeSnapshot`, and restore PVC are **consumer-owned**
and live in the consumer-cluster repo, not in this catalog component. The live
end-to-end snapshot â†’ restore round-trip is a **consumer-cluster E2E test** (it
requires a running cluster with an active CSI driver) and is **not** verified in
this repo.

## Capability

api-surface-only, **no capability** â€” `capabilities: []`. The snapshot-controller
is the sole SIG-Storage reference implementation of the `snapshot.storage.k8s.io`
API; there is no swappable alternative, so no capability id is defined. This is a
permanent design state, not a deferral (so no `# TODO:` in `compatibility.yaml`).
